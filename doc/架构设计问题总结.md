# 架构设计问题总结与解决方案

> **日期**: 2026-02-14  
> **目的**: 总结之前架构设计中的问题，避免再次发生

---

## 一、问题回顾

### 问题 1：Auth 服务设计不合理

**原设计**：
```
前端 → Gateway → Auth 服务 → User 服务 → 数据库
```

**问题**：
1. **性能差**：每个请求都要调用 Auth 服务，增加一次网络往返
2. **职责重叠**：Auth 调用 User 验证，Auth 的存在意义不大
3. **单点故障**：Auth 挂了，整个系统不可用
4. **循环依赖**：Auth 依赖 User，User 也可能依赖 Auth

**正确设计**：
```
前端 → Gateway（本地验证Token） → 业务服务
```

---

### 问题 2：Gateway 缺少认证过滤器

**原设计**：
- Gateway 只做路由转发
- 认证交给 Auth 服务处理

**问题**：
- 每个业务服务都要自己处理认证
- 或者每个请求都要调用 Auth

**正确设计**：
- Gateway 添加全局认证过滤器
- 在 Gateway 层验证 Token（读 Redis）
- 将 userId 通过 Header 传递给下游服务

---

### 问题 3：Feign 返回值不统一

**原设计**：
```java
// 有的返回原始对象
UserDTO getUserById(Long id);

// 有的返回 Result
Result<UserDTO> getUserById(Long id);
```

**问题**：
- 调用方不知道如何处理返回值
- 错误处理不统一

**正确设计**：
```java
// 统一返回 Result
Result<UserDTO> getUserById(Long id);
```

## 二、正确的架构设计原则

### 原则 1：Gateway 统一认证

```
┌─────────────────────────────────────────────────────────┐
│                       Gateway                           │
│                                                         │
│  1. 检查白名单 → 白名单路径直接放行                       │
│  2. 获取 Token → 从 Header 获取 satoken                 │
│  3. 验证 Token → 从 Redis 读取，本地验证（不调用服务）    │
│  4. 提取用户ID → 放入 X-User-Id Header                  │
│  5. 转发请求 → 后端服务从 Header 获取 userId            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 原则 2：服务职责单一

| 服务 | 职责 | 不做的事 |
|------|------|---------|
| Gateway | 路由、认证、限流 | ❌ 业务逻辑 |
| User | 用户管理、登录 | ❌ 商品、订单 |
| Product | 商品管理 | ❌ 用户、订单 |
| Trade | 交易管理 | ❌ 用户、商品 |

### 原则 3：数据隔离

```
User Service  → market_user 数据库（独占）
Product Service → market_product 数据库（独占）
Trade Service → market_trade 数据库（独占）

❌ 禁止：Trade 直接查询 market_user 数据库
✅ 正确：Trade 通过 Feign 调用 User Service
```

### 原则 4：单向依赖

```
依赖关系（允许）：
Trade → Product → User
  │                 ↑
  └─────────────────┘

禁止反向依赖：
User ❌→ Product
User ❌→ Trade
Product ❌→ Trade
```

## 三、正确的认证流程

### 登录流程

```
1. 前端发送登录请求
   POST /api/user/login
   Body: { username, password }

2. Gateway 检查白名单
   /api/user/login 在白名单中，直接放行

3. User Service 处理登录
   - 验证用户名密码
   - 生成 Token（Sa-Token）
   - Token 自动存入 Redis

4. 返回 Token 给前端
   { token: "xxx", userId: 1, username: "xxx" }

5. 前端保存 Token
   localStorage.setItem('satoken', token)
```

### 访问受保护资源

```
1. 前端发送请求
   GET /api/product/my
   Header: satoken: xxx

2. Gateway 认证过滤器
   - 检查白名单：/api/product/my 不在白名单
   - 获取 Token：从 Header 获取
   - 验证 Token：从 Redis 读取验证（本地操作）
   - 提取 userId：放入 X-User-Id Header

3. Product Service 处理请求
   @GetMapping("/my")
   public Result<List<Product>> my(@RequestHeader("X-User-Id") Long userId) {
       return productService.getByUserId(userId);
   }

4. 返回数据给前端
```

## 四、常见错误避免

### ❌ 错误 1：每个服务都验证 Token

```java
// 错误做法：每个服务都这样写
@GetMapping("/my")
public Result<List<Product>> my() {
    if (!StpUtil.isLogin()) {
        throw new BusinessException("未登录");
    }
    Long userId = StpUtil.getLoginIdAsLong();
    // ...
}
```

**✅ 正确做法**：Gateway 统一验证，服务从 Header 获取

```java
@GetMapping("/my")
public Result<List<Product>> my(@RequestHeader("X-User-Id") Long userId) {
    // Gateway 已验证，这里直接使用 userId
    return productService.getByUserId(userId);
}
```

### ❌ 错误 2：服务间传递 Token

```java
// 错误做法：Feign 调用时传递 Token
@GetMapping("/user/{id}")
Result<UserDTO> getUser(@PathVariable Long id, @RequestHeader("satoken") String token);
```

**✅ 正确做法**：Feign 是内部调用，不需要 Token

```java
// 正确做法：内部接口不需要认证
@GetMapping("/feign/user/{id}")
Result<UserDTO> getUser(@PathVariable Long id);
```

### ❌ 错误 3：Auth 服务调用 User 服务

```java
// 错误做法：Auth 通过 Feign 调用 User
@Service
public class AuthServiceImpl {
    @Autowired
    private UserFeignClient userFeignClient;
    
    public LoginVO login(LoginRequest request) {
        Result<UserDTO> result = userFeignClient.validateLogin(...);
        // ...
    }
}
```

**✅ 正确做法**：User 服务直接处理登录

```java
// 正确做法：User 服务自己处理
@Service
public class UserServiceImpl {
    public LoginVO login(LoginRequest request) {
        User user = userMapper.selectByUsername(request.getUsername());
        // 验证密码、生成 Token
        StpUtil.login(user.getId());
        return LoginVO.builder().token(StpUtil.getTokenValue()).build();
    }
}
```

## 五、架构决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 认证位置 | Gateway | 统一入口，避免重复验证 |
| Token 验证 | 本地 Redis | 性能好，无网络开销 |
| Auth 服务 | 不需要 | 职责可合并到 User |
| Feign 返回值 | Result<T> | 统一格式，便于处理 |
| 用户ID传递 | HTTP Header | 简单可靠 |

## 六、检查清单

在设计新功能时，检查以下几点：

- [ ] 是否需要单独的服务？还是可以合并？
- [ ] 服务间是否有循环依赖？
- [ ] 认证是否在 Gateway 处理？
- [ ] Feign 接口返回值是否用 Result 包装？
- [ ] 是否有跨库查询？应该改为 Feign 调用
- [ ] 白名单配置是否正确？

---

## 七、总结

### 核心原则

1. **Gateway 统一认证** - 不要在每个服务中验证
2. **服务职责单一** - 不要让一个服务做太多事
3. **数据隔离** - 每个服务只访问自己的数据库
4. **Feign 统一返回** - 都用 Result<T> 包装
5. **简单可落地** - 不要过度设计

### 最终架构

```
前端
  ↓
Gateway (认证 + 路由)
  ↓
┌─────────────────────────────┐
│  User    Product    Trade   │
│  8101    8102       8103    │
└─────────────────────────────┘
  ↓         ↓          ↓
market_   market_    market_
user      product    trade
```

---

> **记住**：好的架构是简单的架构，能解决问题的最简单方案就是最好的方案。
