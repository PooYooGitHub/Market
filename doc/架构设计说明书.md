## 一、设计原则

### 1.1 核心原则

在设计本架构时，遵循以下原则：

1. **简单可落地** - 毕业设计项目，不过度设计，确保能在有限时间内完成
2. **职责单一** - 每个服务只做一件事，避免职责混乱
3. **数据隔离** - 每个服务独享自己的数据库，禁止跨库访问
4. **统一入口** - 所有请求通过 Gateway 进入，统一认证鉴权
5. **最小依赖** - 服务间调用尽量减少，降低耦合度

### 1.2 关键决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 认证方式 | Gateway 统一认证 | 避免每个请求都调用 Auth 服务，性能更好 |
| Token 存储 | Redis 共享 | 所有服务共享 Redis，实现分布式 Session |
| 服务间调用 | Feign + Result 封装 | 统一返回格式，便于处理 |
| 数据库 | 每服务独立数据库 | 符合微服务数据隔离原则 |

### 2.3 关于 Auth 服务的说明

**⚠️ 重要决策：不单独部署 Auth 服务**

原因：
1. **性能问题** - 如果每个请求都要先调用 Auth 服务验证，会增加网络延迟
2. **单点故障** - Auth 服务挂了，整个系统不可用
3. **职责重复** - Auth 和 User 服务功能重叠

**解决方案**：
- **登录/注册** → 由 User 服务处理（生成 Token）
- **Token 验证** → 由 Gateway 本地验证（读取 Redis）
- **权限校验** → 由 Gateway 统一处理

## 五、服务间调用规范

### 5.1 调用原则

```
┌─────────────────────────────────────────────────────────────────┐
│                     服务间调用规范                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 单向依赖，禁止循环调用                                        │
│                                                                 │
│     Trade ──调用──▶ Product ──调用──▶ User                      │
│       │                                  ▲                      │
│       └──────────────调用────────────────┘                      │
│                                                                 │
│     ✅ 允许：Trade 调用 User                                     │
│     ✅ 允许：Trade 调用 Product                                  │
│     ✅ 允许：Product 调用 User                                   │
│     ❌ 禁止：User 调用 Trade（反向依赖）                          │
│     ❌ 禁止：User 调用 Product（反向依赖）                        │
│                                                                 │
│  2. 返回值必须用 Result<T> 包装                                  │
│                                                                 │
│     // Feign 接口定义                                            │
│     @GetMapping("/{id}")                                        │
│     Result<UserDTO> getUserById(@PathVariable Long id);         │
│                                                                 │
│     // 实现类                                                    │
│     @GetMapping("/{id}")                                        │
│     public Result<UserDTO> getUserById(@PathVariable Long id) { │
│         return Result.success(userService.getById(id));         │
│     }                                                           │
│                                                                 │
│  3. Feign 接口单独放在 Controller 中                             │
│                                                                 │
│     UserController.java      # 前端调用的接口                    │
│     UserFeignController.java # 其他服务 Feign 调用的接口          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 Feign 接口示例

```java
// market-api-user/src/main/java/.../feign/UserFeignClient.java
@FeignClient(name = "market-service-user", path = "/feign/user")
public interface UserFeignClient {
    
    @GetMapping("/{id}")
    Result<UserDTO> getUserById(@PathVariable("id") Long id);
    
    @GetMapping("/batch")
    Result<List<UserDTO>> getUserByIds(@RequestParam("ids") List<Long> ids);
}

// market-service-user/.../controller/UserFeignController.java
@RestController
@RequestMapping("/feign/user")
public class UserFeignController {
    
    @GetMapping("/{id}")
    public Result<UserDTO> getUserById(@PathVariable Long id) {
        UserEntity user = userService.getById(id);
        return Result.success(convertToDTO(user));
    }
}
```

### 6.3 关联数据处理

当需要关联其他服务的数据时，通过 Feign 调用：

```java
// 订单列表需要显示商品信息
public List<OrderVO> getOrderList(Long userId) {
    // 1. 查询订单
    List<Order> orders = orderMapper.selectByUserId(userId);
    
    // 2. 收集商品ID
    List<Long> productIds = orders.stream()
        .map(Order::getProductId)
        .collect(Collectors.toList());
    
    // 3. 批量查询商品信息（Feign 调用）
    Result<List<ProductDTO>> result = productFeignClient.getByIds(productIds);
    Map<Long, ProductDTO> productMap = result.getData().stream()
        .collect(Collectors.toMap(ProductDTO::getId, p -> p));
    
    // 4. 组装数据
    return orders.stream().map(order -> {
        OrderVO vo = new OrderVO();
        BeanUtils.copyProperties(order, vo);
        vo.setProduct(productMap.get(order.getProductId()));
        return vo;
    }).collect(Collectors.toList());
}
```

## 十、FAQ

### Q1: 为什么不单独部署 Auth 服务？

**A**: 单独的 Auth 服务会导致：
1. 每个请求都要调用 Auth 验证，增加延迟
2. Auth 服务成为单点故障
3. 和 User 服务功能重叠

**解决方案**：Gateway 本地验证 Token（读 Redis），User 服务负责登录和生成 Token。

### Q2: 后端服务如何获取当前登录用户？

**A**: Gateway 验证 Token 后，将 userId 放入 HTTP Header：
```java
// Gateway 过滤器
request.mutate().header("X-User-Id", userId.toString());

// 后端 Controller
@GetMapping("/my-orders")
public Result<List<Order>> myOrders(@RequestHeader("X-User-Id") Long userId) {
    return orderService.getByUserId(userId);
}
```

### Q3: 游客可以访问哪些接口？
