# ✅ 文件上传跨域问题解决方案

## 🐛 问题描述

前端上传图片时报错：

```
Access to XMLHttpRequest at 'http://localhost:9000/api/file/upload/avatar' 
from origin 'http://localhost:5173' has been blocked by CORS policy: 
Response to preflight request doesn't pass access control check: 
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

## 🔍 问题分析

### 什么是 CORS？

**CORS (Cross-Origin Resource Sharing)** = 跨域资源共享

当前端和后端运行在不同的域名/端口时，浏览器会阻止跨域请求：

```
前端: http://localhost:5173  (Vue 开发服务器)
后端: http://localhost:9000  (Gateway)
      ↓
结果: 跨域！浏览器拦截！
```

### 为什么会有 CORS？

- **浏览器安全机制** - 防止恶意网站窃取数据
- **同源策略** - 只允许访问同域名、同端口、同协议的资源
- **需要服务器授权** - 服务器必须明确允许跨域访问

## ✅ 解决方案

### 方案：在 File 服务中添加 CORS 配置

已创建 `CorsConfig.java` 文件，配置允许跨域访问。

### 配置内容

```java
@Configuration
public class CorsConfig {

    @Bean
    public CorsFilter corsFilter() {
        CorsConfiguration config = new CorsConfiguration();
        
        // 允许所有域名跨域
        config.addAllowedOriginPattern("*");
        
        // 允许所有请求头
        config.addAllowedHeader("*");
        
        // 允许所有请求方法
        config.addAllowedMethod("*");
        
        // 允许携带认证信息
        config.setAllowCredentials(true);
        
        // 预检请求的有效期
        config.setMaxAge(3600L);
        
        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);
        
        return new CorsFilter(source);
    }
}
```

## 📊 CORS 工作原理

### 简单请求 vs 预检请求

#### 简单请求（Simple Request）
```
GET /api/user/info
直接发送 → 服务器响应（带 CORS 头）
```

#### 预检请求（Preflight Request）
```
POST /api/file/upload
    ↓
1. OPTIONS 请求（预检）
    ← 服务器返回允许的方法、头部
    ↓
2. 真正的 POST 请求
    ← 服务器返回数据
```

**文件上传使用 `multipart/form-data`，会触发预检请求！**

### CORS 响应头

服务器需要返回以下头部：

```http
Access-Control-Allow-Origin: http://localhost:5173
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Content-Type, satoken, Authorization
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 3600
```

## 📝 完整的请求流程

### 成功的文件上传流程

```
1. 用户点击上传
   ↓
2. 前端发送 OPTIONS 请求（预检）
   http://localhost:9000/api/file/upload/avatar
   ↓
3. Gateway 返回 CORS 头
   Access-Control-Allow-Origin: http://localhost:5173
   Access-Control-Allow-Methods: POST, OPTIONS
   ↓
4. 浏览器通过预检，发送真正的 POST 请求
   Headers: { 
     satoken: xxx,
     Content-Type: multipart/form-data 
   }
   Body: FormData { file: ... }
   ↓
5. Gateway 验证 Token → 路由到 File 服务
   ↓
6. File 服务上传到 MinIO
   ↓
7. 返回图片 URL
   {
     "code": 200,
     "data": "http://localhost:9000/market-avatar/xxx.jpg"
   }
   ↓
8. 前端显示上传成功
```

## 🎯 Gateway vs File 服务的 CORS

### 为什么两个地方都要配置？

| 位置 | 作用 | 说明 |
|------|------|------|
| Gateway | 处理路由级别的 CORS | 全局配置，适用于所有服务 |
| File 服务 | 处理服务级别的 CORS | 确保文件上传请求正确响应 |

### 配置冲突？

不会冲突！两者可以共存：
- Gateway 的 CORS 处理路由转发
- File 服务的 CORS 处理实际请求

## 🎓 扩展知识

### 生产环境配置

开发环境使用 `allowedOriginPatterns: "*"` 允许所有域名，但**生产环境应该指定具体域名**：

```java
// 开发环境
config.addAllowedOriginPattern("*");

// 生产环境（推荐）
config.addAllowedOrigin("https://www.your-domain.com");
config.addAllowedOrigin("https://admin.your-domain.com");
```

### 为什么不能用 `*` + `allowCredentials: true`？

```java
// ❌ 错误配置
config.addAllowedOrigin("*");
config.setAllowCredentials(true);
// 浏览器会报错！

// ✅ 正确配置
config.addAllowedOriginPattern("*");  // 或具体域名
config.setAllowCredentials(true);
```

### 常见 CORS 错误

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| No 'Access-Control-Allow-Origin' header | 服务器未返回 CORS 头 | 添加 CORS 配置 |
| CORS policy: credentials mode is 'include' | Origin 不能为 `*` | 使用 `allowedOriginPattern` |
| Method not allowed | 未允许该 HTTP 方法 | 添加 `allowedMethod("*")` |
